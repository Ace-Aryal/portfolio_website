````mdx
# Git Workflow for Junior Developers üöÄ

This guide outlines the **real-world Git workflow** you‚Äôll follow as a junior dev working in teams. It starts from **cloning a repo**, and covers day-to-day practices, including **pulling**, **rebasing**, **committing**, **handling rejected pull requests**, and **avoiding merge conflicts**.

---

## üß≠ Step 1: Clone the Project

```bash
git clone https://github.com/org/project-name.git
cd project-name
```
````

> üîç Make sure you're in the correct working directory (`cd project-name`).

---

## üå± Step 2: Start From a Fresh `main` Branch

```bash
git checkout main
git pull origin main
```

> ‚úÖ **Always pull the latest `main` before starting new work.** This avoids surprises later.

---

## üåø Step 3: Create a Feature Branch

```bash
git checkout -b feat/user-login
```

> ‚úçÔ∏è **Naming Convention:**

- `feat/` ‚Üí new feature (`feat/navbar`)
- `fix/` ‚Üí bug fix (`fix/form-validation`)
- `refactor/` ‚Üí cleanups (`refactor/home-component`)
- `chore/` ‚Üí config or minor updates

---

## üíª Step 4: Work Locally & Commit Often

```bash
git add .
git commit -m "Add login form UI with basic validation"
```

### ‚úÖ Commit Best Practices

- Write clear, **atomic** commits (small, focused)
- Use **present tense**: "Add X", not "Added X"
- Avoid meaningless messages: `Update`, `Fix stuff`

> üìå **Commit frequency**: After every small, testable milestone ‚Äî even if it‚Äôs just styling or a hook.

---

## üîÅ Step 5: Rebase Before You Push

Before pushing, rebase your feature branch with the latest `main` to prevent conflicts and keep a clean history.

```bash
git pull origin main --rebase
```

> üîÑ Why not `merge`? Rebase avoids cluttered commit history full of merge commits.

---

## üöÄ Step 6: Push to Remote

```bash
git push origin feat/user-login
```

Then go to GitHub/GitLab and open a Pull Request.

---

## üìã Step 7: Open a Pull Request (PR)

1. Go to the repository
2. Compare your branch with `main`
3. Add a **clear title** and **description**
4. Tag reviewers (your team lead or senior dev)

> ‚úÖ Describe what the PR does, and how to test it if needed.

---

## ‚ùå Step 8: Rejected PR? Handle Gracefully

If your PR is **rejected or needs changes**:

1. Make the requested edits
2. Commit again
3. Push again ‚Äî the PR will **update automatically**

```bash
git add .
git commit -m "Fix: change form validation as per review"
git push
```

> üîÅ Don‚Äôt take PR reviews personally ‚Äî they‚Äôre about improving code, not criticizing you.

---

## üîÄ Step 9: Merge & Cleanup

Once approved and merged (by you or a team member):

```bash
git checkout main
git pull origin main
git branch -d feat/user-login
git push origin --delete feat/user-login
```

> üßπ Clean up local and remote branches to keep the repo tidy.

---

## ‚ö†Ô∏è Common Merge Conflict Triggers

- You and another dev edited the **same line** in a file.
- You both added different packages at the same time (causing `package-lock.json` conflicts).
- You didn‚Äôt pull latest changes before pushing.
- You auto-formatted files differently (Prettier, ESLint).

### ‚úÖ Conflict Tips

- Use `git status` to see which files are conflicted
- Resolve manually or using VS Code
- After resolving:

```bash
git add .
git rebase --continue
```

---

## üß† Git Best Practices Summary

| Best Practice                       | Why It Matters                   |
| ----------------------------------- | -------------------------------- |
| Start every task from latest `main` | Reduces merge conflicts          |
| Create feature branches per task    | Keeps work isolated, reviewable  |
| Commit small and meaningful chunks  | Easier to understand and debug   |
| Rebase before push                  | Maintains clean, linear history  |
| Push early and get feedback         | Prevents large, unreviewable PRs |
| Handle rejected PRs constructively  | It's part of the dev process     |
| Clean up branches after merge       | Keeps your Git clean             |
